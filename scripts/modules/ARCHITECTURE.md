# Архитектура проекта: Лучшие практики

В этом документе описаны ключевые архитектурные решения и лучшие практики, использованные в проекте. Эти принципы делают код поддерживаемым, масштабируемым и понятным.

---

## 1. Модульность (ES6 Modules)

Код разбит на небольшие, слабо связанные файлы, каждый из которых отвечает за конкретную функциональность.

**Преимущества:**
- Использование `import`/`export` позволяет легко ориентироваться в проекте
- Компоненты можно переиспользовать в разных частях приложения
- Каждый модуль можно тестировать изолированно
- Упрощается командная разработка (разные люди работают с разными файлами)

**Пример:**
```javascript
// math-utils.js
export function add(a, b) {
    return a + b;
}

// main.js
import { add } from './modules/math-utils.js';
console.log(add(2, 3)); // 5
```

## 2. Централизованное управление состоянием (Store + Observer)

Класс `Store` хранит всё состояние приложения и предоставляет методы для его обновления (`setState`) и подписки на изменения (`subscribe`).

Это реализация паттерна **Наблюдатель (Observer)**:
- Компоненты подписываются на изменения состояния
- Перерисовка происходит только при необходимости
- Состояние остаётся **единственным источником правды** (Single Source of Truth)

**Схема работы:**
```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ Компоненты  │────▶│     Store    │────▶│ Уведомление │
│ (подписка)  │◀────│ (состояние)  │◀────│  изменений  │
└─────────────┘     └──────────────┘     └─────────────┘
```

## 3. Неизменяемость данных (Immutability)

Все изменения состояния выполняются через создание **новых** массивов и объектов, а не модификацию существующих.

**Используемые техники:**
- Оператор расширения (`...`)
- Методы `map`, `filter`, `reduce`
- `Object.assign()` для копирования объектов

**Почему это важно:**
```javascript
// ❌ ПЛОХО: мутируем исходный объект
state.todos.push(newTodo);

// ✅ ХОРОШО: создаём новый массив
state.todos = [...state.todos, newTodo];
```

**Преимущества:**
- Предотвращает побочные эффекты
- Упрощает отслеживание изменений (debugging, time-travel)
- Делает код более предсказуемым

---

## 4. Разделение ответственности (Separation of Concerns)

Каждый модуль имеет чёткую зону ответственности:

| Модуль | Ответственность |
|--------|----------------|
| `Store` | Только хранение состояния и уведомление подписчиков |
| `Actions` | Бизнес-логика (добавление, удаление, обновление) |
| `Компоненты` | Только представление (UI) и обработка событий пользователя |
| `Утилиты (db.js)` | Работа с внешними хранилищами (IndexedDB, localStorage) |

**Результат:**
- Легко найти нужный код
- Упрощается поддержка и развитие
- Изменения в одном модуле минимально влияют на другие

---

## 5. Однонаправленный поток данных (Unidirectional Data Flow)

Данные движутся по строго определённому циклу:

```
Действие пользователя 
       ↓
   Вызов action
       ↓
  Обновление store
       ↓
Уведомление подписчиков
       ↓
 Перерисовка компонентов
```

**Преимущества:**
- Поток легко отслеживать и отлаживать
- Исключает несогласованность состояния
- Упрощает понимание того, как данные меняются в приложении

## 6. Компонентный подход (Component-Based UI)

Интерфейс разбит на переиспользуемые компоненты (например: `App`, `AddTodo`, `TodoList`, `TodoItem`).

**Характеристики компонента:**
- Получает необходимые данные через аргументы (props)
- Возвращает готовый DOM-элемент
- Минимально зависит от глобального контекста
- Инкапсулирует свою логику и стили

**Пример:**
```javascript
function TodoItem({ todo, onToggle, onDelete }) {
    const element = document.createElement('div');
    element.className = 'todo-item';
    element.innerHTML = `
        <input type="checkbox" ${todo.completed ? 'checked' : ''}>
        <span>${todo.text}</span>
        <button>Удалить</button>
    `;
    // ... обработчики событий
    return element;
}
```

## 7. Инкапсуляция с помощью Custom Elements и Shadow DOM

Компонент `TodoItem` реализован как пользовательский элемент `<todo-item>`.

**Shadow DOM** обеспечивает:
- Изоляцию стилей компонента от глобальных CSS-правил
- Защиту внутренней структуры от внешнего вмешательства
- Предотвращение конфликтов имён классов и идентификаторов

**Пример:**
```javascript
class TodoItemElement extends HTMLElement {
    constructor() {
        super();
        this.attachShadow({ mode: 'open' });
        this.shadowRoot.innerHTML = `
            <style>
                /* Стили видны только внутри компонента */
                .todo { color: blue; }
            </style>
            <div class="todo">Задача</div>
        `;
    }
}
customElements.define('todo-item', TodoItemElement);
```

## 8. Асинхронность и работа с хранилищем

**IndexedDB** — асинхронное, транзакционное хранилище браузера.

**Реализация:**
- Все вызовы IndexedDB обёрнуты в промисы
- Организована асинхронная инициализация приложения (`async/await`)
- Подписка на изменения `store` автоматически сохраняет состояние в БД

**Преимущества:**
- Гарантированная персистентность данных (сохранение между перезагрузками)
- Нет необходимости в ручных вызовах сохранения
- Приложение работает быстро благодаря асинхронности

## 9. Валидация и обработка ошибок

**Валидация ввода:**
- Проверка на пустую строку
- Проверка максимальной длины
- Отображение ошибок пользователю в реальном времени

**Обработка ошибок:**
- Логирование проблем с базой данных в консоль
- Graceful degradation (приложение не падает, а сообщает об ошибке)
- Пользовательские сообщения об ошибках

**Результат:**
- Повышенная надёжность приложения
- Улучшенный пользовательский опыт (UX)

## 10. Анимации для улучшения UX

**Типы анимаций:**
1. **Появление** новых задач — плавная анимация `fadeIn` (CSS keyframes)
2. **Удаление** задачи — анимация исчезновения (`opacity`, `translateX`)
3. Физическое удаление из DOM происходит **после** завершения анимации

**Пример CSS:**
```css
@keyframes fadeIn {
    from { opacity: 0; transform: translateY(-10px); }
    to { opacity: 1; transform: translateY(0); }
}

.todo-item {
    animation: fadeIn 0.3s ease-out;
}
```

**Преимущества:**
- Интерфейс становится отзывчивым и приятным
- Пользователь визуально понимает, что произошло
- Снижается когнитивная нагрузка

## 💡 Комментарии и рекомендации от разработчика

### Почему именно такая архитектура?

Данная архитектура выбрана не случайно — она представляет собой **золотую середину** между простотой и масштабируемостью:

1. **Не слишком сложная** (как Redux + Saga + Reselect)
2. **Не слишком простая** (как глобальные переменные)
3. **Идеально подходит** для проектов среднего размера (до 10-20 тысяч строк кода)

### Когда использовать этот подход?

| Размер проекта | Рекомендуемая архитектура |
|---------------|--------------------------|
| 🟢 Маленький (< 1000 строк) | Простые модули, без Store |
| 🟡 Средний (1000-10000 строк) | **Эта архитектура** (Store + Observer) |
| 🔴 Крупный (> 10000 строк) | Redux/MobX/Vuex + TypeScript |

### Критические замечания

#### ⚠️ Проблема производительности при частых обновлениях

Паттерн Observer может стать узким местом, если:
- Состояние обновляется очень часто (например, 60 раз в секунду)
- Много подписчиков получают уведомления

**Решение:**
```javascript
// Debounce уведомлений
class Store {
    constructor() {
        this.notifyTimeout = null;
    }
    
    setState(newState) {
        this.state = { ...this.state, ...newState };
        
        // Группируем уведомления (не чаще 1 раза в 16мс ~ 60fps)
        if (!this.notifyTimeout) {
            this.notifyTimeout = setTimeout(() => {
                this.subscribers.forEach(cb => cb(this.state));
                this.notifyTimeout = null;
            }, 16);
        }
    }
}
```

#### ⚠️ Отсутствие типизации

В текущей реализации нет проверки типов на этапе компиляции. Это может привести к ошибкам:

```javascript
// Ошибка будет обнаружена только в runtime
store.setState({ todos: "not an array" }); // ❌
```

**Рекомендация:** Для больших проектов добавьте **TypeScript** или **JSDoc**:

```javascript
/**
 * @typedef {Object} Todo
 * @property {number} id
 * @property {string} text
 * @property {boolean} completed
 */

/**
 * @param {Todo[]} todos
 */
function setTodos(todos) {
    // Теперь IDE подскажет типы
}
```

#### ⚠️ Риск утечек памяти

Если компоненты не отписываются от Store при удалении, возникает утечка памяти:

```javascript
// ❌ ПЛОХО: подписка без отписки
component.mount(() => {
    store.subscribe(component.render);
});

// ✅ ХОРОШО: всегда отписываемся
component.mount(() => {
    const unsubscribe = store.subscribe(component.render);
    component.unmount(() => unsubscribe());
});
```

### Альтернативные подходы

#### 1. Redux (более строгий подход)

**Плюсы:**
- Предсказуемые изменения через reducer'ы
- DevTools для отладки (time-travel debugging)
- Огромная экосистема

**Минусы:**
- Много шаблонного кода (boilerplate)
- Крутая кривая обучения

#### 2. MobX (более простой подход)

**Плюсы:**
- Минимум кода
- Автоматическое отслеживание зависимостей

**Минусы:**
- "Магия" может скрывать проблемы
- Сложнее отлаживать в больших проектах

#### 3. Signals (современный тренд)

**Плюсы:**
- Очень высокая производительность
- Тонкая гранулярность обновлений

**Минусы:**
- Новая концепция, меньше ресурсов для обучения

### Наш выбор: почему Observer + Store?

Для нашего проекта мы выбрали **Observer + Store**, потому что:

1. ✅ **Простота понимания** — новички быстро разбираются
2. ✅ **Минимум зависимостей** — нет внешних библиотек
3. ✅ **Гибкость** — легко адаптировать под любые нужды
4. ✅ **Производительность** — достаточно быстро для наших задач
5. ✅ **Обучающая ценность** — показывает принципы работы фреймворков

### Советы по развитию архитектуры

#### 📈 Если проект растёт:

1. **Добавьте TypeScript** — для типизации данных
2. **Внедрите селекторы** — для оптимизации вычислений:
   ```javascript
   const getCompletedTodos = (state) => 
       state.todos.filter(t => t.completed);
   ```
3. **Добавьте middleware** — для логирования, аналитики:
   ```javascript
   store.use(loggerMiddleware);
   store.use(analyticsMiddleware);
   ```
4. **Разделите Store на модули** — если он становится слишком большим:
   ```javascript
   const store = createStore({
       todos: todoReducer,
       user: userReducer,
       settings: settingsReducer
   });
   ```

#### 🛡️ Для повышения надёжности:

1. **Добавьте валидацию состояния**:
   ```javascript
   function validateState(state) {
       if (!Array.isArray(state.todos)) {
           throw new Error('todos must be an array');
       }
   }
   ```

2. **Внедрите обработку ошибок в actions**:
   ```javascript
   async function addTodo(text) {
       try {
           const todo = await api.createTodo(text);
           store.setState({ todos: [...store.state.todos, todo] });
       } catch (error) {
           store.setState({ error: error.message });
           logger.error(error);
       }
   }
   ```

3. **Добавьте snapshot'ы для отладки**:
   ```javascript
   // Сохраняем историю состояний
   const history = [];
   store.subscribe((state) => {
       history.push(JSON.parse(JSON.stringify(state)));
   });
   ```
