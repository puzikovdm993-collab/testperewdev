# Принципы разработки проекта

Этот документ определяет **обязательные принципы и правила**, на которых строится вся архитектура проекта. 
Каждый разработчик должен следовать этим принципам при написании нового кода.

---

## 🎯 Миссия архитектуры

Создать **поддерживаемое, масштабируемое и понятное** приложение, используя проверенные индустриальные практики без излишнего усложнения.

---

## 12 фундаментальных принципов

### Принцип 1: Модульность превыше всего

**Правило:** Каждый файл отвечает только за одну конкретную задачу.

✅ **Делай:**
```javascript
// math-utils.js — только математические функции
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }
```

❌ **Не делай:**
```javascript
// utils.js — всё подряд (математика, работа с DOM, API запросы)
export function add() {}
export function renderDOM() {}
export function fetchAPI() {}
```

**Критерий качества:** Если название файла не описывает его содержимое однозначно — файл нужно разделить.

---

### Принцип 2: Единый источник правды (Single Source of Truth)

**Правило:** Всё состояние приложения хранится только в `Store`. Компоненты не хранят своё состояние самостоятельно.

✅ **Делай:**
```javascript
// Компонент получает данные из Store
const state = store.getState();
render(state.todos);
```

❌ **Не делай:**
```javascript
// Компонент хранит свою копию данных
let localTodos = [];
function updateTodos(newData) {
    localTodos = newData; // ❌ Вторая копия состояния
}
```

---

### Принцип 3: Неизменяемость данных (Immutability)

**Правило:** Никогда не изменяй существующие объекты и массивы. Создавай новые.

✅ **Делай:**
```javascript
// Создание нового массива
state.todos = [...state.todos, newTodo];
state.todos = state.todos.filter(t => t.id !== id);
state.todos = state.todos.map(t => 
    t.id === id ? { ...t, completed: true } : t
);
```

❌ **Не делай:**
```javascript
// Прямая мутация
state.todos.push(newTodo);           // ❌
state.todos.splice(index, 1);        // ❌
state.todos[0].completed = true;     // ❌
delete state.todos[0];               // ❌
```

**Инструменты:** Используй `...`, `map`, `filter`, `reduce`, `Object.assign()`.

---

### Принцип 4: Чёткое разделение ответственности

**Правило:** Каждый модуль знает только о своей зоне ответственности.

| Модуль | Может | Не может |
|--------|-------|----------|
| **Store** | Хранить данные, уведомлять подписчиков | Знать о UI, делать запросы к API |
| **Actions** | Изменять Store, вызывать API | Работать с DOM напрямую |
| **Components** | Рендерить UI, отправлять события | Изменять Store напрямую |
| **Utils** | Предоставлять вспомогательные функции | Знать о бизнес-логике |

**Схема зависимостей:**
```
Components → Actions → Store → Utils
     ↓          ↓         ↓
   Events    State     Notifications
```

---

### Принцип 5: Однонаправленный поток данных

**Правило:** Данные движутся только в одном направлении.

```
┌─────────────────┐
│  Пользователь   │
└────────┬────────┘
         │ Клик/Ввод
         ▼
┌─────────────────┐
│   Component     │ ──► Dispatch Action
└─────────────────┘
         │
         ▼
┌─────────────────┐
│    Action       │ ──► Обновляет Store
└─────────────────┘
         │
         ▼
┌─────────────────┐
│     Store       │ ──► Уведомляет подписчиков
└─────────────────┘
         │
         ▼
┌─────────────────┐
│   Components    │ ──► Перерисовка
└─────────────────┘
```

❌ **Запрещено:**
- Прямое изменение Store из компонентов
- Обратный поток (Component ← Store без Action)
- Циклические зависимости

---

### Принцип 6: Компонентный подход

**Правило:** UI состоит из независимых, переиспользуемых компонентов.

**Требования к компоненту:**
1. Получает данные через параметры (props)
2. Возвращает DOM-элемент
3. Не зависит от глобального состояния напрямую
4. Инкапсулирует свою логику

✅ **Пример:**
```javascript
function TodoItem({ todo, onToggle, onDelete }) {
    const element = document.createElement('div');
    element.className = 'todo-item';
    
    // Используем только переданные данные
    element.innerHTML = `
        <input type="checkbox" ${todo.completed ? 'checked' : ''}>
        <span>${todo.text}</span>
        <button onclick="${onDelete}">Удалить</button>
    `;
    
    return element;
}
```

---

### Принцип 7: Инкапсуляция через Shadow DOM

**Правило:** Стили и структура компонента должны быть изолированы.

✅ **Делай:**
```javascript
class TodoItem extends HTMLElement {
    constructor() {
        super();
        this.shadow = this.attachShadow({ mode: 'open' });
        this.shadow.innerHTML = `
            <style>
                /* Стили видны только внутри */
                .btn { color: red; }
            </style>
            <button class="btn">Click</button>
        `;
    }
}
```

**Преимущества:**
- Нет конфликтов имён классов
- Глобальные CSS не влияют на компонент
- Стили компонента не «протекают» наружу

---

### Принцип 8: Асинхронность с async/await

**Правило:** Все асинхронные операции используют `async/await`, не `.then()` цепочки.

✅ **Делай:**
```javascript
async function loadTodos() {
    try {
        const todos = await db.getAll();
        store.setState({ todos });
    } catch (error) {
        console.error('Failed to load:', error);
        store.setState({ error: error.message });
    }
}
```

❌ **Не делай:**
```javascript
// Устаревший стиль
db.getAll()
    .then(todos => store.setState({ todos }))
    .catch(error => console.error(error));
```

**Требование:** Обработка ошибок обязательна через `try/catch`.

---

### Принцип 9: Валидация на границах системы

**Правило:** Проверяй все входящие данные перед использованием.

**Границы для проверки:**
1. Ввод пользователя (формы, инпуты)
2. Ответы от API / базы данных
3. Параметры функций (в критичных местах)

✅ **Пример:**
```javascript
function addTodo(text) {
    // Валидация ввода
    if (!text || typeof text !== 'string') {
        throw new Error('Invalid todo text');
    }
    if (text.trim().length === 0) {
        throw new Error('Todo text cannot be empty');
    }
    if (text.length > 500) {
        throw new Error('Todo text is too long');
    }
    
    // Дальше работаем с данными
    const todo = { id: Date.now(), text: text.trim() };
    // ...
}
```

### Принцип 10: UX через анимации

**Правило:** Все значимые изменения UI сопровождаются анимацией.

**Обязательные анимации:**
- Появление новых элементов (`fadeIn`)
- Удаление элементов (`fadeOut` + сдвиг)
- Изменение состояния (например, завершение задачи)

✅ **Пример CSS:**
```css
@keyframes fadeIn {
    from { opacity: 0; transform: translateY(-10px); }
    to { opacity: 1; transform: translateY(0); }
}

.todo-item {
    animation: fadeIn 0.3s ease-out;
}

.todo-item.removing {
    transition: opacity 0.3s, transform 0.3s;
    opacity: 0;
    transform: translateX(100%);
}
```

**Важно:** Физическое удаление из DOM происходит **после** завершения анимации.

---

### Принцип 11: Современный JavaScript (ES6+)

**Правило:** Используй только современные возможности языка.

**Обязательный набор:**
- ✅ Стрелочные функции `() => {}`
- ✅ Классы `class Name {}`
- ✅ Деструктуризация `const { a, b } = obj`
- ✅ Шаблонные строки `` `Hello ${name}` ``
- ✅ Методы массивов `map`, `filter`, `reduce`
- ✅ Async/await для асинхронности
- ✅ Модули `import`/`export`
- ✅ Spread/Rest операторы `...`

❌ **Запрещено:**
- `var` (только `let` и `const`)
- Function declaration для обычных функций
- Конкатенация строк через `+`
- Циклы `for` там, где можно использовать методы массивов

### Принцип 12: Прогрессивное улучшение

**Правило:** Приложение должно работать максимально хорошо в текущих условиях.

**Стратегия:**
1. Определяем поддержку возможностей браузера
2. Используем современные фичи если доступны
3. Предоставляем fallback для старых браузеров
4. Показываем понятные сообщения при критических проблемах

✅ **Пример:**
```javascript
// Проверка поддержки IndexedDB
if (!window.indexedDB) {
    console.warn('IndexedDB не поддерживается');
    // Переключаемся на localStorage или показываем предупреждение
    useLocalStorageFallback();
}

// Проверка поддержки Custom Elements
if (!customElements) {
    // Используем обычные div вместо веб-компонентов
    renderWithDivs();
}
```

## 🚫 Запрещённые приёмы (Black List)

Следующие приёмы **категорически запрещены** в проекте:

| Приём | Почему запрещено | Альтернатива |
|-------|-----------------|--------------|
| `var` | Проблемы с областью видимости | `let`, `const` |
| Прямая мутация состояния | Побочные эффекты, баги | Создание новых объектов |
| Глобальные переменные | Конфликты имён, сложность тестирования | Модули с export |
| jQuery | Избыточность, устаревший подход | Нативный DOM API |
| `.then()` цепочки | Менее читаемый код | `async/await` |
| `document.write` | Блокирует рендеринг, устарело | `innerHTML`, `createElement` |
| `eval()` | Безопасность, производительность | Динамический `import()` |
| Магические числа | Непонятный код | Именованные константы |
| Большие функции (>50 строк) | Сложно понимать и тестировать | Разделение на подфункции |
| Компоненты >200 строк | Нарушение SRP | Разбиение на подкомпоненты |

## ✅ Чек-лист для код-ревью

Перед тем как сдать код, проверь:

### Структура и модульность
- [ ] Код разбит на логические модули
- [ ] Каждый файл имеет одну ответственность
- [ ] Нет циклических зависимостей между модулями
- [ ] Использованы `import`/`export`

### Работа с состоянием
- [ ] Состояние изменяется только через Store
- [ ] Нет прямой мутации объектов/массивов
- [ ] Созданы новые копии при изменениях
- [ ] Подписчики отписываются при уничтожении

### Компоненты
- [ ] Компоненты получают данные через параметры
- [ ] Нет зависимости от глобального состояния
- [ ] Стили инкапсулированы (Shadow DOM или префиксы)
- [ ] Компоненты переиспользуемы

### Асинхронность
- [ ] Используется `async/await`
- [ ] Есть обработка ошибок (`try/catch`)
- [ ] Нет race conditions
- [ ] Данные загружаются до рендеринга

### Качество кода
- [ ] Нет `var`, только `let`/`const`
- [ ] Нет магических чисел (есть константы)
- [ ] Функции короткие (<50 строк)
- [ ] Понятные имена переменных и функций
- [ ] Есть комментарии для сложной логики

### UX
- [ ] Есть валидация ввода
- [ ] Показаны сообщения об ошибках
- [ ] Добавлены анимации для изменений
- [ ] Интерфейс отзывчивый

### Совместимость
- [ ] Есть проверка поддержки фич
- [ ] Предусмотрены fallback'и
- [ ] Сообщения для старых браузеров

## 🔄 Процесс внесения изменений

При добавлении новой функциональности следуй алгоритму:

### Шаг 1: Анализ
1. Определить, какой модуль отвечает за эту область
2. Проверить, не нарушит ли изменение существующую логику
3. Решить, нужен ли новый модуль или можно расширить существующий

### Шаг 2: Проектирование
1. Спроектировать интерфейс нового модуля (export'ы)
2. Определить зависимости от других модулей
3. Продумать обработку ошибок

### Шаг 3: Реализация
1. Написать код следуя принципам выше
2. Добавить обработку ошибок
3. Добавить анимации (если влияет на UI)
4. Написать комментарии для сложной логики

### Шаг 4: Тестирование
1. Проверить работу в разных сценариях
2. Проверить обработку ошибок
3. Проверить производительность
4. Пройти чек-лист для код-ревью

### Шаг 5: Документирование
1. Обновить README модуля (если нужно)
2. Добавить примеры использования
3. Описать ограничения (если есть)

## 🎓 Контрольные вопросы

Проверь своё понимание принципов:

<details>
<summary><b>1. Можно ли изменить массив todos напрямую через push()?</b></summary>

❌ **Нет.** Нужно создать новый массив:
```javascript
state.todos = [...state.todos, newTodo];
```
</details>

<details>
<summary><b>2. Где должно храниться состояние приложения?</b></summary>

✅ **Только в Store.** Компоненты получают данные из Store и не хранят свои копии.
</details>

<details>
<summary><b>3. Какой синтаксис использовать для асинхронного кода?</b></summary>

✅ **async/await.** Цепочки `.then()` запрещены.
</details>

<details>
<summary><b>4. Что делать, если функция получилась больше 50 строк?</b></summary>

✅ **Разбить на несколько меньших функций.** Каждая должна решать одну подзадачу.
</details>

<details>
<summary><b>5. Можно ли компоненту напрямую вызвать store.setState()?</b></summary>

❌ **Нет.** Компонент должен dispatch'ить action, который уже обновит Store.
</details>
